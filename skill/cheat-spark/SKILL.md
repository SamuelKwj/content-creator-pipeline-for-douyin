# /cheat-spark — 火花固化

灵感捕获的第一道门。把模糊的感觉变成可追溯、可打分、可排队的内容资产。

---

## 触发词

| 用户说 | 模式 | AI 做什么 |
|--------|------|----------|
| "我有个想法"/"灵感来了"/"火花" | **火花收录** | 给 3-4 标题选项 → 等选 → 盲打分 → 写入 inbox.jsonl |
| "记下来"/"先记下来" | **火花收录** | 同上 |
| "归入档案"/"这就是该火花文章" | **口述后归档** | 把已整理的文章正式收录为火花 |
| "我要口述"/"我会发语音"/"我说你记" | **口述采集** | 进入采集模式，只转录暂存，不等说完不整理 |
| "说完了"/"完事了，整理" | **口述整理** | 通篇整理所有暂存片段 |
| "把这句加入 X 作为金句卡" | **金句追加** | 不收录，直接 patch 到目标文章的金句备选区 |

## 一、火花收录（标准流程）

### 铁律

> **绝不跳过标题选项直接收录。** 灵感不能裸写进档案——必须经过「标题化」这个提炼步骤，让创作者自己确认角度。

### 流程

```
用户: "我有个想法：XXX"
  ↓
AI: 提炼核心洞察，给 3-4 个标题选项
    - 每个选项代表一个不同的切入角度
    - 标题要有传播力、有钩子、像个人说的
  ↓
用户: 选一个，或改一个
  ↓
AI: 盲打分（delegate_task → cheat-score-blind sub-agent）
    只传 spark 文本 + rubric_notes.md
  ↓
AI: 写入 archive/inbox.jsonl
    包含：id, title, core_insight, spark_raw, score, created_at, stage="spark"
  ↓
AI: 告知用户分数 + 收录完成
```

### 标题选项要求

- **3-4 个**，每个 ≤30 字
- 每个选项代表**不同的切入角度**（不是同一角度的措辞变体）
- 有钩子、有人味、不 AI 腔
- 如果用户给了方向偏好，围绕那个方向展开选项

### inbox.jsonl 条目格式

```json
{
  "id": "spark-YYYYMMDD-001",
  "title": "用户确认的标题",
  "core_insight": "一句话核心洞察",
  "spark_raw": "用户原始灵感文本",
  "title_options": ["选项1", "选项2", "选项3"],
  "chosen_index": 0,
  "rubric_score": {"dim1": 3, "dim2": 4, ...},
  "composite": 3.5,
  "created_at": "2026-07-02T10:30:00",
  "stage": "spark",
  "status": "pending",
  "tags": []
}
```

### 注意事项

- **系列/栏目/长期企划**不当普通火花处理：写成系列企划文档，归档到 `archive/<series-name>_系列企划.md`，写入 `archive/topics.jsonl`
- 只有**单条可拍内容**才进 `archive/inbox.jsonl`
- 收录后提示用户下一步：`cheat-seed` 深挖写稿

## 二、口述采集模式

用户要口述一篇文章/想法，分多段语音或文字输入。

### 进入条件

用户说"我要口述"、"我会一直发语音"、"你先识别，最后我说'说完了'再整理"、"我说你记"。

### 采集阶段规则

1. **立即进入采集模式**，不把第一句当成正式灵感收录
2. 每条后续语音/文字只做：**转录 → 去水词 → 暂存片段编号**
3. 输出要短：`已暂存片段 N` + 去水词后的片段文本
4. **不要通篇整理、不要改观点、不要升华**
5. 如果用户说"接下来，不是记下来"——表示他要启动未来采集流程，不是让你把这句写进档案
6. 用户说"说完了"或"完事了，整理" → 进入整理阶段

### 整理阶段规则

1. 把所有暂存片段通篇整理成连贯文章
2. 只去水词、理顺逻辑、补过渡——**不加新观点、不扩写、不改结论**
3. 整理完成后**仍不自动收录为火花**
4. 输出整理结果，等用户确认

### 归档触发

只有用户再说"归入档案"或"这就是该火花文章"——才进入火花收录流程：

1. 给 3-4 标题选项
2. 用户选/改
3. 盲打分
4. 写入 inbox.jsonl

---

## 三、金句追加（不入库）

用户说"把这句加入 X 文章作为金句卡片"时：

1. 这是**按需生产**，不是新火花——不收录、不盲打分
2. 定位目标文章/脚本，patch 到金句备选区
3. 附用途标记如 `【金句卡片 / 中段转场】`
4. 不要求标题选项

## 四、Common Pitfalls

### Pitfall 1: 跳过标题选项直接收录
- **症状**：用户给了灵感，AI 直接写进 inbox.jsonl
- **修复**：灵感必须经过标题化提炼，让创作者确认角度。这是铁律。

### Pitfall 2: 口述第一句就当灵感收录
- **症状**：用户说"我要口述"，AI 把第一句当成正式火花打分了
- **修复**：先判断是否进入采集模式。口述采集模式下不做任何收录动作。

### Pitfall 3: 整理完自动归档
- **症状**：口述整理完成后 AI 自动写入 inbox.jsonl
- **修复**：整理完只是输出结果，必须等用户说"归入档案"才归档。

### Pitfall 4: 标题选项角度雷同
- **症状**：3-4 个选项只是同一角度的措辞变化
- **修复**：每个选项必须代表不同的切入角度。如果灵感本身角度单一，诚实说"这个灵感角度比较单一，只有 X 方向的发挥空间"，给出 2 个选项即可。

### Pitfall 5: 忘记盲打分
- **症状**：收录火花时 AI 自己 inline 打分
- **修复**：所有评分必须通过 `delegate_task` 调 `cheat-score-blind` sub-agent。火花收录也不例外。
